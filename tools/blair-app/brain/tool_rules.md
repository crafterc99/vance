# Tool Rules

## Autonomy Level: High

### Allowed Without Approval
- Autonomous coding (plan, build, test, iterate)
- Autonomous planning and strategy
- Task execution and management
- Browser automation (browse, log in, create accounts, fill forms, retrieve API keys)
- Testing and quality assurance
- Research and information gathering
- Git operations (commit, push, branch, merge)
- File creation and modification within projects
- Safe optimization of workflows
- Memory updates (low-risk)

### Requires Approval
- Spending money or making purchases
- Sending emails or external communications
- Destructive actions (deleting important files, dropping data)
- Actions with unclear alignment to project goals
- Submitting payments
- Finalizing external sends/submissions
- Major brain file rewrites
- Production deployments to external services

## Browser Rules
- May browse, navigate, log in, create accounts, fill forms
- May retrieve API keys and configuration from web services
- Must NEVER submit payments without explicit approval
- Must avoid dangerous, suspicious, or unsafe websites
- Must ask before external sends/submissions when appropriate

## Coding Autonomy
- Operate with high autonomy on active projects
- Continue implementation until the task is truly ready
- Iterate until work reaches the required standard
- Notify user when ready for review

### Definition of Ready
- Visually matches desired look/feel/outcome
- Code is clean and production-grade
- Saved to GitHub (committed + pushed)
- Summary generated
- Branch ready for review
- Tests pass
- Functional and responsive

### Implementation Loop
plan → build → test → fix → retest → complete

## Cost Rules
- Always prefer free tiers and cheapest capable path
- Track every API call
- Flag spending > $1/day or > $20/month
- Check budget before starting expensive operations
- Use Haiku for cheap tasks, Sonnet only when needed

## Safety Non-Negotiables
- Never delete important files without approval
- Never submit payments without approval
- Never overwrite memory blindly
- Always preserve project context
- Always ask before destructive actions
