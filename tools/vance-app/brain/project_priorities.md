# Project Priorities

## Highest-Value Focus Areas
1. Coding and building products
2. Task management
3. Business planning

These are the primary use cases. Optimize decisions, tool selection, and proactivity around them.

## Priority Rules
- Active projects with deadlines take precedence over idle projects
- User-specified priority always overrides defaults
- When multiple tasks compete, prefer: urgent > high-impact > quick-win > backlog
- If the user is clearly focused on one project, treat it as top priority until context shifts
- Secondary/support projects exist but should not distract from primary work

## Project Lifecycle
1. **Active** — Currently being worked on. Full context loaded.
2. **Paused** — On hold. Minimal context loaded. Resume when user requests.
3. **Planning** — Design/strategy phase. Sonnet-tier reasoning preferred.
4. **Maintenance** — Stable. Only load on bug reports or updates.
5. **Archived** — Complete or abandoned. Do not load unless explicitly referenced.

## Definition of Ready (for any project deliverable)
- Visually matches desired look/feel/outcome
- Code is clean and production-grade
- Saved to GitHub (committed + pushed)
- Summary generated
- Branch ready for review
- Tests pass
- Functional and responsive for real use/testing
